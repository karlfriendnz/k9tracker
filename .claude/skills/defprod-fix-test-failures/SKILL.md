# Fix Test Failures

Reads a test check report (generated by `/defprod-run-area-tests`) and implements the recommended fix for each failed or flaky test. Test faults get test code fixes; production faults get production code fixes.

## When to use

- After `/defprod-run-area-tests` has produced a report with failures.
- When invoked via `/defprod-fix-test-failures <area-key>`.

## Inputs

- **Area key** — the area whose test failures to fix (e.g. `AUTH`, `BILLING`).

## Config

This skill consults `.defprod/defprod.json` for optional hints. If the file doesn't exist, the skill discovers everything automatically.

| Key | Type | Purpose |
|-----|------|---------|
| `products[].name` | `string` | Matches product name to locate config entry |
| `products[].e2eDir` | `string` | Path to the e2e test directory — used to locate the report |
| `products[].compileCheck` | `string` | Command to verify compilation after changes |

---

## Workflow

### Step 1 — Load the report

Find and read the test check report:
- Look in `e2eDir` from `.defprod/defprod.json` if available
- Search for `<AREA-KEY>-test-check.md` in the test directory

Parse the summary table and detail sections. Identify all items classified as test faults and production faults.

If no report is found, tell the user and suggest running `/defprod-run-area-tests` first.

---

### Step 2 — Fix test faults

Process each test fault from the report:

1. **Read the test file** at the location specified in the report.
2. **Apply the recommended fix** — update selectors, assertions, timing waits, setup logic, or test data as needed.
3. **Run the individual test** to confirm the fix works.
4. If the test still fails after the fix, re-analyse:
   - Was the report's classification correct?
   - Is there an additional issue?
   - Fix iteratively until the test passes.

Common test fault fixes:
- **Stale selectors** — update `data-testid`, role queries, or text matchers to match current markup
- **Timing issues** — replace arbitrary waits with framework-provided waiting mechanisms
- **Outdated assertions** — update expected values to match current behaviour
- **Setup/teardown** — fix test data creation or cleanup steps

---

### Step 3 — Fix production faults

Process each production fault from the report:

1. **Read the failing test** to understand what behaviour is expected.
2. **Read the relevant user story** — call `getUserStory` from the DefProd MCP server to confirm the expected behaviour matches the acceptance criteria.
3. **Read the production code** that the test exercises.
4. **Implement the fix** — modify the production code so it satisfies the acceptance criteria and passes the test.
5. **Verify compilation** — if the project has a compile/build step, run it and fix any errors.
6. **Run the test** to confirm the fix works.

For production faults, ensure the fix:
- Satisfies the acceptance criteria from the user story
- Doesn't break other tests
- Follows the project's coding conventions

---

### Step 4 — Full area re-run

After fixing all individual items, run the entire area's test suite to catch any regressions or interactions between fixes.

If new failures appear:
1. Analyse and fix them.
2. Re-run until the full suite passes.

---

### Step 5 — Final compilation check

If any production code was changed, run the project's compilation/build check one final time to confirm everything is clean. Use `compileCheck` from `.defprod/defprod.json` if available.

---

### Step 6 — Update report

Append a "Fixes Applied" section to the existing test check report:

```markdown
## Fixes Applied — <YYYY-MM-DD>

| # | Test | Fault | Fix | Files Changed |
|---|------|-------|-----|---------------|
| 1 | test name | Test | Updated selector to match new data-testid | file.spec.ts |
| 2 | test name | Production | Fixed validation logic to reject empty input | validator.ts, validator.spec.ts |

**Final result:** {N}/{N} tests passing
```

---

### Step 7 — Output summary

> **Fixes applied for {Area Name}:**
> - {N} test faults fixed
> - {N} production faults fixed
> - {N} files changed
> - **Final result:** {N}/{N} tests passing
>
> Report updated at `<path>/<AREA-KEY>-test-check.md`

---

## Rules

- **Fix in the right layer** — test faults are fixed in test code only. Production faults are fixed in production code. Don't "fix" a production bug by changing the test to expect wrong behaviour.
- **Verify each fix individually** — run the specific test after each fix before moving to the next.
- **Re-run the full suite** — individual fixes can interact. The full re-run catches regressions.
- **Respect the user story** — when fixing production faults, the acceptance criteria are the source of truth for expected behaviour. If the test and the story disagree, the story wins.
- **Don't skip the compilation check** — if you changed production code, compilation must pass before you're done.
- **Update the report** — the report is a living document. Append fixes so there's a record of what was done.
