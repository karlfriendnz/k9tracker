# Fix Discrepancies

Reads a discrepancy analysis report (generated by `/defprod-analyze-discrepancies`) and implements the recommended changes — updates user stories via the DefProd MCP server, modifies codebase where needed, and verifies the results.

## When to use

- After `/defprod-analyze-discrepancies` has produced a report with findings.
- When invoked via `/defprod-fix-discrepancies <report-path>`.

## Inputs

- **Report path** — path to the discrepancy report file (e.g. `docs/defprod-discrepancy-AUTH-2026-04-05.md`).

## Config

This skill consults `.defprod/defprod.json` for optional hints. If the file doesn't exist, the skill discovers everything automatically.

| Key | Type | Purpose |
|-----|------|---------|
| `products[].name` | `string` | Matches product name to locate config entry |
| `products[].frontendApp` | `string` | Path to the frontend app |
| `products[].backendApp` | `string` | Path to the backend app |
| `products[].e2eDir` | `string` | Path to e2e test directory |
| `products[].compileCheck` | `string` | Command to verify compilation |

## Workflow

### Step 1 — Load and parse the report

Read the discrepancy report. Parse all four tables:
1. Stories/ACs with no implementation
2. ACs with incomplete implementation
3. Implemented features with no story
4. Stories that diverge from implementation

Count the total items and present a summary to the user:

> **Discrepancy report loaded:**
> - {N} stories/ACs with no implementation → will update/remove from definition
> - {N} partially implemented ACs → will update definition or fix code
> - {N} implemented features with no story → will create stories
> - {N} diverging stories → will align definition with code
>
> Proceed with all fixes, or would you like to cherry-pick?

Wait for the user's go-ahead before proceeding.

---

### Step 2 — Verify MCP availability

1. Call `listAreas` from the DefProd MCP server. Identify the area from the report.
2. Call `listUserStories` for the area. Load current stories to ensure the report is still relevant.

If stories have changed since the report was generated, flag any items that may be outdated.

---

### Step 3 — Apply definition changes

Process the report's recommendations that involve updating the product definition:

#### Table 1 items (no implementation):
- If recommendation is "Remove from definition": call `patchUserStory` to remove the unimplemented acceptance criteria, or `deleteUserStory` if the entire story is unimplemented.
- If recommendation is "Implement": skip — this will be handled as a code change in Step 4.

#### Table 2 items (incomplete implementation):
- If recommendation is "Update definition": call `patchUserStory` to update acceptance criteria to match what's actually implemented.
- If recommendation is "Fix code": skip — handled in Step 4.

#### Table 3 items (no story):
- Call `createUserStory` for each implemented feature that should have a story. Set `status: completed`.

#### Table 4 items (diverged details):
- If recommendation is "Update definition": call `patchUserStory` to align the story with the actual implementation.
- If recommendation is "Fix code": skip — handled in Step 4.

---

### Step 4 — Apply codebase changes

Process recommendations that require code changes:

#### Table 1 items (recommendation: "Implement"):
- Implement the missing feature to satisfy the acceptance criteria.
- Follow the project's coding conventions and patterns.

#### Table 2 items (recommendation: "Fix code"):
- Modify the production code to fully satisfy the acceptance criteria.

#### Table 4 items (recommendation: "Fix code"):
- Update the production code to match what the story describes.

After each code change:
- Verify compilation passes (use `compileCheck` from config if available)
- Run any related tests to confirm the change doesn't break anything

---

### Step 5 — Verify

For each story that was modified or created:

#### 5a. Check acceptance criteria

Verify that every acceptance criterion on the updated/new story is satisfied by the codebase. If the project has a running dev server, use browser automation to verify UI changes.

#### 5b. Update e2e tests

If the area has e2e tests:
- **Test exists and passes** — no action
- **Test exists but needs updating** — update to cover changed criteria, run until green
- **No test for a new story** — create one covering all acceptance criteria, run until green

Use `e2eDir` from config if available.

---

### Step 6 — Summary

Present a complete summary:

> **Discrepancy fixes applied for {Area Name}:**
>
> **Definition changes:**
> - {N} stories updated
> - {N} stories created
> - {N} stories deleted
>
> **Codebase changes:**
> - {N} files modified
> - {list of changed files}
>
> **Tests:**
> - {N} tests updated
> - {N} tests created
> - All passing: Yes/No
>
> **Items not applied** (if any):
> - [item] — [reason]

## Rules

- **Always get user confirmation** before starting fixes. The user may want to cherry-pick which items to fix.
- **Definition changes first, code changes second** — update stories before implementing code so the code changes can be verified against the updated criteria.
- **Don't invent beyond the report** — fix what the report recommends. If you discover additional issues while fixing, note them for a future analysis run rather than expanding scope.
- **Verify compilation after every code change** — don't batch code changes and hope they all compile.
- **New stories are `completed`** — stories created for existing features should have `status: completed`.
- **Respect the report's recommendations** — if the report says "update definition", update the definition. If it says "fix code", fix the code. Don't swap the approach unless the user explicitly requests it.
